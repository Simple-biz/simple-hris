## 3. What was changed in code

### 3.1 RBAC-only authorisation for disputes (#4)

**File**: `src/lib/supabase/pab-day-disputes.ts`

Removed `DISPUTE_APPROVERS` constant and `isDisputeApprover` helper. `canActOnDisputes(email)` is now a single `SELECT role FROM employee_roles WHERE LOWER(work_email) = email AND revoked_at IS NULL` with a role membership check against `DISPUTE_ACTOR_ROLES` (`payroll_coordinator`, `payroll_manager`, `finance`, `hr_coordinator`, `admin`).

**Operational implication**: Carla and Fran must have active `employee_roles` rows — otherwise they lose dispute access the instant this ships. Verify with:

```sql
SELECT work_email, role, revoked_at FROM employee_roles
WHERE LOWER(work_email) IN ('carla@simple.biz', 'franm@simple.biz');
```

### 3.2 Orphanage "day after" forgiveness (#3)

Preserves a single DB row per visit (simple data model, simple admin-delete semantics) and extends forgiveness at read time.

**`src/components/PayrollWizard.tsx`** — after building the primary `approvedDisputeDates` map, a second loop walks approved disputes and, for `reason === 'orphanage_visit'`, sets `forgivenDates[visit_date + 1] = null` **unless that date already has an entry** (real dispute wins). A new `addOneIsoDay(iso)` helper does string-safe date arithmetic via `Date.UTC`.

**`src/components/employee/EmployeeDashboard.tsx`** — `disputesByDate` useMemo does the same for the employee calendar: synthesises a cloned `PabDayDisputeRow` at `visit_date + 1` with `override_hours = null` so the existing per-cell "forgiven" check picks it up without special-casing.

**Net effect**: approved orphanage visits now forgive both the visit date and the day after with a 4h floor (original Payroll intent, now matches UI copy at `OrphanageVisits.tsx:225` and `:431`). Non-orphanage reasons remain single-date.

### 3.3 Audit log role resolution (#5)

New helpers `fetchActiveRoles(email)` and `resolveUserRole(email, fallback)` in `pab-day-disputes.ts`. Every `insertAuditLog` call in the dispute lib was wrapped in an async IIFE that looks up the real role from `employee_roles` (priority-ordered: `admin` → `finance` → `payroll_manager` → …) before writing. Falls back to the caller-supplied default (`Employee` for submissions/withdrawals, `Admin` for decisions) when the email has no active role rows.

Affected audit actions: `pab_dispute.submitted`, `pab_dispute.approved`, `pab_dispute.denied`, `pab_dispute.edited`, `pab_dispute.withdrawn`.

### 3.4 Atomic orphanage visit upsert (#6)

**File**: `src/lib/supabase/pab-day-disputes.ts` — `adminCreateOrphanageVisit`

Replaced the `SELECT id → INSERT or UPDATE` pattern with `.upsert(payload, { onConflict: 'work_email,dispute_date' }).select('id').single()`. Concurrent inserts for the same `(work_email, dispute_date)` no longer race; the unique index handles resolution atomically.

### 3.5 `override_hours` tri-state (#7)

**Contract finalised**:

| Stored value | Meaning on `dispute_date` |
|---|---|
| `NULL` | No override. Hubstaff hours stand. 4h floor-drop applies. |
| `0` | Intentional zero-out. Day counts as 0h. Fails PAB. |
| `> 0` | SET. Replaces Hubstaff hours for this day. |

**Server** (`decideDispute`, `editDisputeDecision`): `params.override_hours != null && params.override_hours >= 0 ? value : null`. Previously `> 0`, which silently collapsed `0`.

**Payroll calc** (`PayrollWizard.tsx` `perfectAttendanceEligible`, `employeeWeekdayHours`): `overrideHours != null ? overrideHours * 3600 : rawSeconds`. Previously `> 0 ? … : rawSeconds`.

**Employee calc** (`EmployeeDashboard.tsx` `pabCalendar`): `if (set == null || set < 0) continue` — 0 is valid and writes 0 seconds into `hoursByDateKey`, flowing through the existing `day.passes` logic to fail the day.

**Client UI** (`PabDisputeQueue.tsx` `handleDecide` and `handleEdit`): sends `null` when both hour+minute inputs are empty strings; otherwise sends the computed total (including 0). Approvers who explicitly type `0h 0m` can now zero-out a day.

### 3.6 Dead code removal (#8)

- `revokeFirstApproval()` function deleted from `src/lib/supabase/pab-day-disputes.ts`.
- `first_approved_by`, `first_approved_at`, `first_approved_note`, `first_approved_override_hours` removed from `PabDayDisputeRow` type.
- `revoke_first` action handler removed from `app/api/pab-disputes/[id]/route.ts`; the API now accepts only `approve`, `deny`, `edit`.
- SQL migration dropped the four DB columns (confirmed run by user on 2026-04-21).

### 3.7 Timezone-safe ISO extraction (#9)

New helper `parseColDateParts(col)` in `PayrollWizard.tsx` that returns `{year, month, day}` directly from the column-name regex, bypassing `Date` entirely. `isoDateFromColumnGroup` now uses it. The existing `parseColDate` (returning a `Date`) is retained for callers that want a Date object for display / weekday derivation — those paths construct and query in local time consistently and weren't actually buggy in practice.

### 3.8 Shared `normEmail` helper (#11)

`src/components/payroll/OrphanageVisits.tsx` now uses `normEmail()` from `@/lib/email/norm-email` in four places where it previously called ad-hoc `.toLowerCase()`: `employeeByEmail` map build, `displayNameFor`, `handleSubmit`, and the search dropdown key. Matches the convention used elsewhere (`PayrollWizard.tsx`, `EmployeeDashboard.tsx`, all of `app/api/*`).

## 4. Database changes

One migration ran on 2026-04-21:

```sql
ALTER TABLE public.pab_day_disputes
  DROP COLUMN IF EXISTS first_approved_by,
  DROP COLUMN IF EXISTS first_approved_at,
  DROP COLUMN IF EXISTS first_approved_note,
  DROP COLUMN IF EXISTS first_approved_override_hours;
```

If the original `ADD COLUMN` migration (listed as pending #8 in `memory/pending_sql.md`) was never run, the `DROP COLUMN IF EXISTS` is a no-op — safe either way.

No data loss: the two-tier flow was never written to by `decideDispute` after 2026-04-18, so the columns were empty or near-empty.
