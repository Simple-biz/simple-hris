## 1. Dashboard hero pulse animation (Overview.tsx)

## 5. Google Sheet sync (manual button-only)

A "Sync from Google Sheet" link sits below each of the master + rates upload buttons. Clicking it pulls the configured Google Sheet via a service-account JWT and feeds it through the same Supabase ingest as the CSV upload path. Documented end-to-end in [csv-imports.md](../features/csv-imports.md). High points:

### Auth

- Hand-rolled service-account JWT (`src/lib/google-sheets/auth.ts`) using Node's built-in `crypto.createSign` — no `googleapis` dep added. Token cached in-memory until 30s before expiry.
- One service account is shared across the master + rates sheets, both shared with that account's email as Viewer (or Editor).

### Fetcher quirks

- `fetch-master-sheet.ts` **auto-detects the header row** (first row containing a `department` cell + `name` or `personal email` cell), drops everything above it, and **prepends two synthetic `MASTERLIST` sentinel rows** so `validateMasterListCsvLayout()` accepts the resulting CSV. Lets a naturally laid out Google Sheet (row 1 = headers) flow through unchanged.
- `fetch-rates-sheet.ts` is simpler — no header detection, no sentinel.
- Both fetchers **wrap the configured tab name in single quotes** before URL-encoding for Sheets API A1 notation. Required when tab names have spaces (e.g. `All Dept`).

### Endpoints

- `POST/GET /api/cron/sync-master-from-sheet`
- `POST/GET /api/cron/sync-rates-from-sheet`

Path segment `cron/` is legacy — no scheduled cron exists. `vercel.json` has `"$schema"` only; no `crons` array. Triggered exclusively by the in-app button. `CRON_SECRET` optional; if set, blocks the in-app button until session-based admin auth is wired as an alternate path (not yet done — left as a follow-up flag in the UI doc).

## 6. Ingest pipeline correctness fixes

Three issues surfaced during the first real-data sync of a 784-row master sheet. All three were latent bugs that would also affect the regular CSV upload path.

### 6.1 Case-sensitive existing-row lookup → duplicate-key INSERT failure

| File | Path |
|---|---|
| `src/lib/supabase/global-master-list-db.ts` | `replaceGlobalMasterListFromCsvText()` |
| `src/lib/supabase/rates-upload-db.ts` | `replaceEmployeeHourlyRatesFromCsv()` |

**Symptom**: `Insert failed (batch 0–50): duplicate key value violates unique constraint "global_master_list_person_dept_key"`.

**Root cause**: the existing-row lookup chunked by `.in('"Personal Email"', emails)` is case-sensitive on the PostgREST side. DB rows from the legacy backfill (and any manual-edit rows) sometimes have mixed-case `Personal Email`. They were invisible to the lookup → CSV's lowercase counterparts queued for INSERT → the partial unique index on `(LOWER("Personal Email"), LOWER("Department"))` rejected them.

**Fix**: replaced the chunked `.in()` with a single full-table SELECT (`.range(0, 9999)`) and folded case in memory. Same change applied to the rates ingest's two-pronged (`Work Email` + `Personal Email`) lookup.

**Trade-off**: full-table SELECT every sync. Acceptable at the system's roster size (≤ a few thousand). Faster than the chunked approach in practice. Would need rethinking if/when the table grows past ~50k rows.

### 6.2 Within-CSV duplicate identity rows → same INSERT failure

**Symptom**: same error as 6.1 but persists even after the case-insensitivity fix when the source CSV/sheet itself contains two rows for the same `(personal_email, department)` pair.

**Root cause**: the dedup logic only checked CSV rows against the DB; rows from the **same** CSV that shared an identity key both missed the existingByKey check (neither in DB yet) and both queued for INSERT. Second INSERT hit the unique index.

**Fix** (master only — rates ingest already dedupes per work email by keeping the latest week): added a `Map<identityKey, row>` step before the partition into update/insert. Last occurrence wins. New return field `duplicatesInCsv` carries the collapsed-row count up to both upstream callers (CSV upload + Google Sheet sync) and the result panel surfaces it as a subline.

### 6.3 3-minute sequential UPDATE loop → parallel chunks

**Symptom**: master sync of ~700 mostly-existing rows took ~3 minutes; rates sync had the same shape latently.

**Root cause**: per-row sequential UPDATE round-trips. ~250ms per Supabase round-trip × 700 rows ≈ 3 min. Would 504 in production (Vercel serverless function timeout).

**Fix**: collected updates into an array, then `Promise.all` chunks of 20 sequentially. Each row touches a distinct primary key — no deadlock concern. New runtime: ~5 sec for 700 rows. Same fix applied to both master and rates ingest.

## 7. Rates page Department chip not displaying

**Symptom**: every card in the Rates / Profiles tab was missing the blue Department chip, even for employees whose master-list row has a real department.

**Root cause**: `mergeSourcesDeduped([rates, master])` in `src/lib/supabase/employee-rate-profiles.ts` kept the first occurrence of each field key. The rates row has `Department: null` (the rates ingest never writes Department), so it claimed the key with a null value, and the master row's actual `"Accounting Team"` was silently dropped.

**Fix**: `mergeSourcesDeduped` now skips `null`, `undefined`, and whitespace-only string values so later sources can fill gaps. Side benefit: same fix surfaces Phone, Address, Organization, Bank Preferred, etc. from the master row when the rates row has them blank.

**Cache caveat**: `getEmployeeRateProfileSummaries` caches results for 60 seconds in module scope. After the fix, restart `npm run dev` (or wait 60s) before hard-refreshing to see the corrected chips.

## 8. Security: GitHub secret scanner caught a service-account JSON in the repo

**Trigger**: pushing the unpushed commit `1da2253` ("Synced!") was blocked by GitHub's secret scanner because it contained `references/global-master-list-hris-48ccf40267e0.json` — a Google Cloud service-account credentials file.

**Resolution path** (in progress at end of session):
1. **Rotate the key in GCP Console** — delete the compromised key from the service-account's Keys tab; create a new key; update `.env.local` and Vercel env vars.
2. `.gitignore` updated with patterns for service-account JSON files (`**/*-iam.gserviceaccount.com.json`, `**/global-master-list-hris-*.json`, `**/service-account-*.json`, `**/*-service-account.json`, `**/credentials*.json`, `references/secrets/`).
3. Move the JSON file out of the repo (or delete) → `git rm --cached <file>` → `git add .gitignore` → `git commit --amend --no-edit` → `git push origin main`.

Step 1 + 3 must be done by the user (key rotation + commit amend). `.gitignore` change is in this commit.
